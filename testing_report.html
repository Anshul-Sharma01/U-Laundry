🔴 Critical Issues 3 Found · 3 Fixed

C-01  CRITICAL  Login OTP Never Sent — Authentication Completely Broken

server/src/controllers/user.controller.js   loginUser()

The sendEmail() call inside loginUser was entirely commented out. The server generated and hashed an OTP, then logged it to the console in plaintext — but never emailed it to the user. This means login was impossible for any user without direct server log access. The plaintext OTP in the console is also a security exposure.

❌ Before
const plainOtp = await Otp.generateOtp(email, 'login', 2);
console.log("otp : ", plainOtp); // OTP leaked to logs!

// await sendEmail(
// user.email, "U-Laundry Login OTP",
// `...${plainOtp}...`
// );
✅ After
const plainOtp = await Otp.generateOtp(email, 'login', 2);

await sendEmail(
user.email,
"U-Laundry Login OTP",
`...${plainOtp}...`
);

C-02  CRITICAL  N+1 Query Bomb in Pickup Slot Builder — 28 Sequential DB Calls

server/src/controllers/order.controller.js   buildAvailablePickupSlots()

buildAvailablePickupSlots() looped over 7 days × 4 slot labels and fired a separate findOneAndUpdate for each combination — 28 sequential round-trips to MongoDB per request. Under any load this causes severe latency and can exhaust the connection pool.

❌ Before — 28 sequential queries
for (let dayOffset = ...) {
for (const slotLabel of PICKUP_SLOT_LABELS) {
const inventory = await PickupSlotInventory
.findOneAndUpdate(...) // ← 1 query per iteration
slots.push(...);
}
}
✅ After — 2 bulk operations total
// 1. Build all slot keys upfront
const slotKeys = [...all 28 pairs...];

// 2. Single bulkWrite to upsert missing slots
await PickupSlotInventory.bulkWrite(bulkOps);

// 3. Single find() to fetch all inventory
const docs = await PickupSlotInventory.find({
slotDate: { $in: slotDates },
slotLabel: { $in: PICKUP_SLOT_LABELS }
}).lean();

C-03  CRITICAL  Chatbot Response Format Inconsistency — API Contract Broken

server/src/controllers/chatbot.controller.js   askChatbot()

Every other endpoint returns ApiResponse with shape { success, statusCode, data, message }. The chatbot returned a raw { success: true, data: { answer } } object — no statusCode, no message. The frontend client expecting data.data.answer would silently receive undefined if the response shape ever changed. Also missing: question length cap (DoS vector).

❌ Before
res.status(200).json({
success: true,
data: { answer: answer.trim() }
});
✅ After
// Length guard added
if (question.trim().length > 1000)
throw new ApiError(400, "...");

return res.status(200).json(
new ApiResponse(200,
{ answer: answer.trim() },
"Response generated successfully"
)
);

🟠 High Priority Issues 5 Found · 5 Fixed

H-01  HIGH  Unrestricted File Upload — No Size Limit, No Type Validation

server/src/middlewares/multer.middleware.js

Multer was configured with no file size limit and no MIME type filter. Any authenticated user could upload a multi-gigabyte file or a non-image (PDF, executable, etc.), exhausting disk space and bypassing the intended image-only constraint.

❌ Before
export const upload = multer({ storage });
✅ After
const fileFilter = (req, file, cb) => {
const allowed = ["image/jpeg","image/png",
"image/webp","image/gif"];
if (allowed.includes(file.mimetype))
cb(null, true);
else cb(new Error("Images only"), false);
};
export const upload = multer({
storage,
limits: { fileSize: 5 * 1024 * 1024, files: 1 },
fileFilter,
});

H-02  HIGH  Fractional Quantity Allowed in Orders

server/src/controllers/order.controller.js   addNewOrder()

Number(item.quantity) >= 1 passes for 1.5, 2.7, etc. A user could order 1.5 shirts, causing incorrect pricing and confusing moderators. Added Number.isInteger() guard.

❌ Before
if (!item.quantity || Number(item.quantity) < 1) {
throw new ApiError(400, "Quantity must be at least 1");
}
✅ After
const qty = Number(item.quantity);
if (!item.quantity || qty < 1) {
throw new ApiError(400, "Quantity must be at least 1");
}
if (!Number.isInteger(qty)) {
throw new ApiError(400, "Quantity must be a whole number");
}

H-03  HIGH  Pagination Parameters Not Clamped — Negative/Zero Values Accepted

server/src/controllers/order.controller.js   getOrdersByUser() · getAllOrders()

parseInt(page) || 1 returns 1 for NaN but passes through -5 or 0 directly, causing negative skip values in MongoDB queries. Also no upper bound on limit — a client could request limit=999999.

❌ Before
page = parseInt(page) || 1;
limit = parseInt(limit) || 5;
✅ After
page = Math.max(1, parseInt(page) || 1);
limit = Math.min(100, Math.max(1, parseInt(limit) || 5));

H-04  HIGH  Moderator Fetches Only Active Items — Inactive Items Invisible

client/src/store/slices/moderatorSlice.ts   fetchAllLaundryItems()

fetchAllLaundryItems called GET /laundry-items (public endpoint, returns only isActive: true items). Moderators managing the catalog would never see deactivated items. The correct endpoint is /laundry-items/admin/all.

❌ Before
const { data } = await axiosInstance.get('laundry-items');
return data.data; // only active items!
✅ After
// Use admin/all to get ALL items including inactive
const { data } = await axiosInstance.get('laundry-items/admin/all');
return data.data;

H-05  HIGH  Admin Slice Uses Wrong Route Prefix for Orders

client/src/store/slices/adminSlice.ts   fetchAllOrders · fetchOrdersByStatus · updateOrderStatus

Three admin thunks used orders/ prefix (plural). The canonical backend mount is /api/v1/order/ (singular). While the server has a backwards-compat alias, relying on it silently is fragile and the alias could be removed. Fixed to use the canonical order/ prefix.

❌ Before
axiosInstance.get(`orders/getall?...`)
axiosInstance.get(`orders/get/${status}`)
axiosInstance.patch(`orders/update/${id}/${s}`)
✅ After
axiosInstance.get(`order/getall?...`)
axiosInstance.get(`order/get/${status}`)
axiosInstance.patch(`order/update/${id}/${s}`)
